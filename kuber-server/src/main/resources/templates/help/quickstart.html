Quick Start

1. Start the Server

# Required JVM options for LMDB persistence on Java 9+
java --add-opens=java.base/java.nio=ALL-UNNAMED \
     --add-opens=java.base/sun.nio.ch=ALL-UNNAMED \
     -jar kuber-server-2.1.0.jar

# Or use the startup script (includes all required JVM options)
./scripts/kuber-start.sh

2. Connect with Redis CLI

redis-cli -p 6380 -a admin123
127.0.0.1:6380> PING
PONG
127.0.0.1:6380> SET user:1001 "John Doe"
OK
127.0.0.1:6380> GET user:1001
"John Doe"
Session Management (v2.1.0): Use CLIENT SETTIMEOUT <seconds> to keep connections alive longer. See Connection Management.

3. Use REST API

# Store a value
curl -X PUT -u admin:admin123 \
  -H "Content-Type: application/json" \
  -d '{"name":"John","status":"active"}' \
  http://localhost:8080/api/cache/default/user:1001

# Get a value
curl -u admin:admin123 http://localhost:8080/api/cache/default/user:1001

4. Access Web UI

Open http://localhost:8080 and login with admin / admin123

Default Ports: REST API on 8080, Redis protocol on 6380

5. Use Messaging (Optional)

Access cache via message brokers. Test clients are provided for all supported brokers:

# Install Python dependencies
pip install kafka-python stomp.py pika

# Run Kafka test client
cd kuber-client-python/examples
python kafka_request_response_test.py

# Run ActiveMQ test client (STOMP on port 61613)
python activemq_request_response_test.py

# Run RabbitMQ test client
python rabbitmq_request_response_test.py
Test Clients Available: Python, Java, and C# clients for Kafka, ActiveMQ, RabbitMQ, and IBM MQ. See Messaging Test Clients for complete examples.

Key Operational Highlights

Memory Management v2.1.0
  • Hybrid Architecture: All keys always in memory, values can overflow to disk
  • Memory Pressure: Auto-evicts when heap exceeds 85%
  • Count-Based Limits: Max 20% of keys or 10,000 values per region
  • No OOM Crashes: Values are persisted before eviction
Key Management
  • KeyIndex: Every key tracked in memory (~100 bytes/key)
  • Off-Heap Option: Zero GC for 10+ million keys
  • Fast Operations: EXISTS, KEYS, DBSIZE are all O(1) or O(n) in memory
  • Never Evicted: Keys remain even under memory pressure
Persistence & Durability
  • Multiple Backends: RocksDB (default), LMDB, SQLite, MongoDB
  • Crash-Safe: RocksDB WAL, LMDB ACID, SQLite journaling
  • Async Writes: Fast response, background persistence
  • Auto-Compaction: RocksDB optimized at 2 AM daily
Data Safety & Recovery
  • Startup Recovery: All data recovered from persistence on restart
  • Orchestrated Startup: Spring → Compaction → Cache → Redis → Autoload
  • No Data Loss: Graceful shutdown completes pending writes
  • TTL Cleanup: Expired entries automatically removed
Event Publishing v1.3.2
  • 5 Publishers: Kafka, ActiveMQ, RabbitMQ, IBM MQ, File
  • Pluggable: Interface-driven, easy to extend
  • Async: Non-blocking, fire-and-forget
  • Per-Region: Different destinations per region
Request/Response Messaging v2.1.0
  • 4 Brokers: Kafka, ActiveMQ, RabbitMQ, IBM MQ
  • Broker Controls: Enable, Disable, Pause, Resume
  • Backpressure: Auto pause/resume on queue depth
  • Batch Ops: MGET/MSET support with limits
  • Test Clients: Python, Java, C# for all brokers
Role-Based Access Control v2.1.0
  • 4 Permissions: READ, WRITE, DELETE, ADMIN
  • Region-Specific: Control access per cache region
  • Hot Reload: No restart needed for security changes
  • Full Enforcement: Web UI, REST API, Redis protocol
Learn More: See System Internals for detailed documentation on memory management, persistence, thread model, and data safety. See Event Publishing for outbound events. See Request/Response Messaging for cache access via message brokers. See Security & RBAC for role-based access control.
