Kuber Help Documentation

Welcome to Kuber Distributed Cache

Kuber is a high-performance, Redis-compatible distributed caching system with:

  • Multi-Backend Persistence - Choose from MongoDB, PostgreSQL, SQLite, RocksDB, or in-memory storage
  • JSON Document Support - Store and query complex JSON with JSONPath
  • Region Partitioning - Logical namespaces for multi-tenant applications
  • Redis Compatibility - Use existing Redis clients and tools
  • High Performance - Sub-millisecond operations with Caffeine cache
  • Replication - Primary/Secondary architecture with ZooKeeper coordination
  • Autoload - Bulk data import from CSV and JSON files

Browse, query, and manage cache entries through this web interface

HTTP API on port 8080 with JSON responses

Native Redis protocol on port 6379

Quick Start

Start Kuber Server 
# Start Kuber Server (uses embedded persistence by default)
java -jar kuber-server-1.1.0-SNAPSHOT.jar

# Or with specific persistence backend
java -jar kuber-server-1.1.0-SNAPSHOT.jar --kuber.persistence.type=sqlite
java -jar kuber-server-1.1.0-SNAPSHOT.jar --kuber.persistence.type=rocksdb
java -jar kuber-server-1.1.0-SNAPSHOT.jar --kuber.persistence.type=memory

# For MongoDB persistence (requires MongoDB server)
mongod --dbpath /data/db
java -jar kuber-server-1.1.0-SNAPSHOT.jar --kuber.persistence.type=mongodb

# With custom ports
java -jar kuber-server-1.1.0-SNAPSHOT.jar --kuber.network.port=6380 --server.port=8081
Using redis-cli

Web UI Guide

The dashboard provides an overview of your Kuber instance:

  • Server Status - Node ID, replication mode (PRIMARY/SECONDARY/STANDALONE)
  • Connection Stats - Active Redis connections, server port
  • Cache Stats - Total entries, max memory entries, persistence mode
  • Regions Overview - List of all regions with entry counts and hit ratios

Browse and manage cache entries:

  • Region Selector - Switch between cache regions
  • Keys Table - List of all keys with pagination and search
  • Export CSV - Download all entries as CSV file
  • Insert Entry - Add new string or JSON entries
  • View Entry - See value, type, TTL, and edit
  • Delete Entry - Remove entries with confirmation
Use the search box to filter keys. Pagination controls let you navigate large result sets.

Execute advanced queries:

Manage cache regions:

  • View Regions - Cards showing entry count, hit ratio, creation date
  • Create Region - Add new region with name and description
  • Region Details - Statistics, operations per second, memory usage
  • Export Region - Download all entries as CSV
  • Purge Region - Delete all entries (Admin only)
  • Delete Region - Remove entire region (Admin only)

Redis Protocol Reference

Port: 6379 (configurable)
Authentication: AUTH password or AUTH username password
Client: Any Redis client (redis-cli, Jedis, redis-py, etc.)

Connection Commands

$ redis-cli -p 6379

# Authenticate (password only)
127.0.0.1:6379> AUTH admin123
OK

# Authenticate (username + password)
127.0.0.1:6379> AUTH admin admin123
OK

# Ping server
127.0.0.1:6379> PING
PONG

# Quit
127.0.0.1:6379> QUIT
OK

String Commands

# SET - Store a string value
127.0.0.1:6379> SET user:1001 "John Doe"
OK

# SET with expiration (EX = seconds, PX = milliseconds)
127.0.0.1:6379> SET session:abc123 "data" EX 3600
OK

# SETEX - SET with expiration in one command
127.0.0.1:6379> SETEX temp:key 60 "expires in 60 seconds"
OK

# SETNX - SET only if key doesn't exist
127.0.0.1:6379> SETNX user:1001 "Won't overwrite"
(integer) 0

# GET - Retrieve a value
127.0.0.1:6379> GET user:1001
"John Doe"

# GETSET - Set new value and return old
127.0.0.1:6379> GETSET counter "100"
"99"

# INCR / DECR - Increment/Decrement integer values
127.0.0.1:6379> SET counter 10
OK
127.0.0.1:6379> INCR counter
(integer) 11
127.0.0.1:6379> DECR counter
(integer) 10
127.0.0.1:6379> INCRBY counter 5
(integer) 15
127.0.0.1:6379> DECRBY counter 3
(integer) 12

# APPEND - Append to existing string
127.0.0.1:6379> APPEND user:1001 " (Admin)"
(integer) 17
127.0.0.1:6379> GET user:1001
"John Doe (Admin)"

# STRLEN - Get string length
127.0.0.1:6379> STRLEN user:1001
(integer) 17

Key Commands

# KEYS - Find keys matching pattern (glob)
127.0.0.1:6379> KEYS user:*
1) "user:1001"
2) "user:1002"
3) "user:1003"

127.0.0.1:6379> KEYS *:active
1) "session:active"
2) "config:active"

# EXISTS - Check if key exists
127.0.0.1:6379> EXISTS user:1001
(integer) 1

# DEL - Delete keys

(integer) 2

# TYPE - Get key type
127.0.0.1:6379> TYPE user:1001

# RENAME - Rename a key
127.0.0.1:6379> RENAME old:key new:key

TTL Commands

# TTL - Get remaining time to live (seconds)
127.0.0.1:6379> TTL session:abc123
(integer) 3540

(integer) -1

127.0.0.1:6379> EXPIRE user:1001 3600
(integer) 1

# PERSIST - Remove TTL (make permanent)
127.0.0.1:6379> PERSIST user:1001
(integer) 1

Python Client Examples

Kuber is fully compatible with standard Redis clients. Use redis-py or the included kuber-client-python package.

Installation

# Using standard redis-py
pip install redis

# Or using Kuber's Python client
pip install kuber-client

Basic Connection and Operations

import redis

# Connect to Kuber
client = redis.Redis(
    host='localhost',
    port=6379,
    password='admin123',  # Or use username='admin', password='admin123'
    decode_responses=True  # Return strings instead of bytes
)

# Test connection
print(client.ping())  # True

# Basic string operations
client.set('user:1001', 'John Doe')
client.set('user:1002', 'Jane Smith')

name = client.get('user:1001')
print(f"User: {name}")  # User: John Doe

# Set with expiration (TTL)
client.setex('session:abc123', 3600, 'session-data')  # Expires in 1 hour
client.set('temp:key', 'value', ex=60)  # Alternative syntax

# Check TTL
ttl = client.ttl('session:abc123')
print(f"TTL: {ttl} seconds")

# Increment/Decrement
client.set('counter', 0)
client.incr('counter')  # 1
client.incrby('counter', 10)  # 11
client.decr('counter')  # 10

# Check existence and delete
if client.exists('user:1001'):
    print("User exists")
    
client.delete('user:1001', 'user:1002')  # Delete multiple keys

Key Pattern Matching

import redis

client = redis.Redis(host='localhost', port=6379, password='admin123', decode_responses=True)

# Create some test data
for i in range(1, 101):
    client.set(f'user:{i}', f'User {i}')
    client.set(f'order:{i}', f'Order {i}')

# Find keys matching pattern (glob syntax)
user_keys = client.keys('user:*')
print(f"Found {len(user_keys)} user keys")

# Find keys with single character wildcard
keys = client.keys('user:?')  # user:1 through user:9

# Find specific patterns
active_keys = client.keys('*:active')
order_keys = client.keys('order:[1-9]*')  # Orders 1-99

# SCAN for large datasets (recommended over KEYS)
cursor = 0
all_keys = []
while True:
    cursor, keys = client.scan(cursor, match='user:*', count=100)
    all_keys.extend(keys)
    if cursor == 0:
        break
print(f"Scanned {len(all_keys)} keys")

Hash Operations

import redis

client = redis.Redis(host='localhost', port=6379, password='admin123', decode_responses=True)

# Store user profile as hash
client.hset('user:1001:profile', mapping={
    'name': 'John Doe',
    'email': 'john@example.com',
    'age': '30',
    'city': 'New York'
})

# Get single field
name = client.hget('user:1001:profile', 'name')
print(f"Name: {name}")

# Get multiple fields
fields = client.hmget('user:1001:profile', ['name', 'email', 'age'])
print(f"Fields: {fields}")  # ['John Doe', 'john@example.com', '30']

# Get all fields
profile = client.hgetall('user:1001:profile')
print(f"Profile: {profile}")
# {'name': 'John Doe', 'email': 'john@example.com', 'age': '30', 'city': 'New York'}

# Update single field
client.hset('user:1001:profile', 'age', '31')

# Check if field exists
if client.hexists('user:1001:profile', 'phone'):
    print("Has phone")
else:
    client.hset('user:1001:profile', 'phone', '+1-555-1234')

# Get field count
field_count = client.hlen('user:1001:profile')
print(f"Fields: {field_count}")

# Delete field
client.hdel('user:1001:profile', 'phone')

# Get all keys or values
keys = client.hkeys('user:1001:profile')
values = client.hvals('user:1001:profile')

JSON Document Operations

import redis
import json

client = redis.Redis(host='localhost', port=6379, password='admin123', decode_responses=True)

# Store JSON using JSET command
user_data = {
    "name": "John Doe",
    "age": 30,
    "email": "john@example.com",
    "address": {
        "city": "New York",
        "zip": "10001"
    },
    "tags": ["developer", "python", "redis"]
}

# Using execute_command for Kuber extensions
client.execute_command('JSET', 'user:1001', '$', json.dumps(user_data))

# Get entire JSON document
result = client.execute_command('JGET', 'user:1001', '$')
data = json.loads(result)
print(f"User: {data['name']}")

# Get specific path
name = client.execute_command('JGET', 'user:1001', '$.name')
city = client.execute_command('JGET', 'user:1001', '$.address.city')
print(f"Name: {name}, City: {city}")

# Search JSON documents
# First, create some test documents
for i in range(1, 11):
    doc = {
        "name": f"User {i}",
        "status": "active" if i % 2 == 0 else "inactive",
        "score": i * 10
    }
    client.execute_command('JSET', f'player:{i}', '$', json.dumps(doc))

# Search for active users
results = client.execute_command('JSEARCH', '$.status=active')
print(f"Active users: {results}")

# Search with multiple conditions
results = client.execute_command('JSEARCH', '$.status=active,$.score>50')
print(f"Active users with score > 50: {results}")

Batch Operations

import redis

client = redis.Redis(host='localhost', port=6379, password='admin123', decode_responses=True)

# MSET - Set multiple keys at once
client.mset({
    'key1': 'value1',
    'key2': 'value2',
    'key3': 'value3'
})

# MGET - Get multiple keys at once
values = client.mget(['key1', 'key2', 'key3', 'nonexistent'])
print(values)  # ['value1', 'value2', 'value3', None]

# Pipeline for atomic operations
pipe = client.pipeline()
pipe.set('counter', 0)
pipe.incr('counter')
pipe.incr('counter')
pipe.incr('counter')
pipe.get('counter')
results = pipe.execute()
print(f"Pipeline results: {results}")  # [True, 1, 2, 3, '3']

# Transaction with MULTI/EXEC
pipe = client.pipeline(transaction=True)
pipe.set('balance:1001', 1000)
pipe.decrby('balance:1001', 100)
pipe.incrby('balance:1002', 100)
pipe.execute()

Region Management

import redis

client = redis.Redis(host='localhost', port=6379, password='admin123', decode_responses=True)

# List all regions
regions = client.execute_command('REGIONS')
print(f"Regions: {regions}")

# Create new region
client.execute_command('RCREATE', 'analytics')

# Get region info
info = client.execute_command('RINFO', 'default')
print(f"Region info: {info}")

# Select region (all subsequent commands use this region)
client.execute_command('RSELECT', 'analytics')
client.set('pageviews', 1000)

# Switch back to default
client.execute_command('RSELECT', 'default')

# Purge region (delete all entries but keep region)
client.execute_command('RPURGE', 'analytics')

# Delete region entirely
client.execute_command('RDROP', 'analytics')

Using Kuber Python Client Package

from kuber import KuberClient

# Create client with region support
client = KuberClient(

    default_region='default'
)

results = client.json_search('$.status=active')
keys = client.ksearch(r'user:\d+')  # Regex search

# Region management
client.create_region('sessions', description='Session storage')
regions = client.list_regions()
client.select_region('sessions')

# Connection pool for production
from kuber import KuberConnectionPool

pool = KuberConnectionPool(
    host='localhost',
    port=6379,
    password='admin123',
    max_connections=10
)

with pool.get_client() as client:
    client.set('key', 'value')

Pipeline and Transactions

import redis.clients.jedis.Jedis;
import redis.clients.jedis.Pipeline;
import redis.clients.jedis.Transaction;
import redis.clients.jedis.Response;

public class KuberPipelineExample {
    public static void main(String[] args) {
        try (Jedis jedis = new Jedis("localhost", 6379)) {
            jedis.auth("admin123");
            
            // Pipeline - multiple commands in one round trip
            Pipeline pipeline = jedis.pipelined();
            pipeline.set("key1", "value1");
            pipeline.set("key2", "value2");
            pipeline.set("key3", "value3");
            Response<String> r1 = pipeline.get("key1");
            Response<String> r2 = pipeline.get("key2");
            pipeline.sync();
            
            System.out.println("Key1: " + r1.get());
            System.out.println("Key2: " + r2.get());
            
            // Transaction - atomic execution
            Transaction tx = jedis.multi();
            tx.set("balance:1001", "1000");
            tx.decrBy("balance:1001", 100);
            tx.incrBy("balance:1002", 100);
            tx.exec();
            
            System.out.println("Balance 1001: " + jedis.get("balance:1001"));
            System.out.println("Balance 1002: " + jedis.get("balance:1002"));
        }
    }
}

Autoload - Bulk Data Import

Autoload enables automatic bulk data import from CSV and JSON files. Simply drop files in the inbox folder and Kuber will load them into the cache.

How It Works

The autoload service watches a configurable directory (default: ./autoload) with two subfolders:

  • inbox/ - Place your data files here along with metadata files
  • outbox/ - Processed files are moved here with timestamps and status

The watcher runs every 60 seconds (configurable) and processes files that have both:

  1. A data file (.csv or .json)
  2. A corresponding metadata file (.csv.metadata or .json.metadata)

Metadata File Format

Each data file must have a metadata file with the same name plus .metadata extension:

# Example: users.csv.metadata
region:users
ttl:3600
key_field:user_id
delimiter:,

# Fields:
# region     - Target region (default: "default")
# ttl        - Time-to-live in seconds (default: -1 = no expiration)
# key_field  - Column/field name to use as cache key (REQUIRED)
# delimiter  - CSV delimiter character (default: ",")

CSV File Format

CSV files must have a header row. Each row is converted to a JSON document using the header fields as keys.

users.csv

user_id,name,email,age,status
1001,John Doe,john@example.com,30,active
1002,Jane Smith,jane@example.com,25,active
1003,Bob Wilson,bob@example.com,35,inactive

users.csv.metadata

region:users
ttl:3600
key_field:user_id

Result: Three JSON entries are created in the "users" region with keys "1001", "1002", "1003":

# Key: 1001
{"user_id": 1001, "name": "John Doe", "email": "john@example.com", "age": 30, "status": "active"}

# Key: 1002
{"user_id": 1002, "name": "Jane Smith", "email": "jane@example.com", "age": 25, "status": "active"}

JSON File Format (JSONL)

JSON files should have one complete JSON object per line (JSON Lines format):

products.json

{"sku":"PROD001","name":"Widget","price":29.99}
{"sku":"PROD002","name":"Gadget","price":49.99}
{"sku":"PROD003","name":"Gizmo","price":19.99}

products.json.metadata

region:products
ttl:-1
key_field:sku

            for (int i = 1; i <= 100; i++) {
                String json = String.format(
                    "{\"order_id\":\"ORD%04d\",\"customer\":\"Customer %d\",\"total\":%.2f}",
                    i, i, Math.random() * 1000
                );
                writer.write(json);
                writer.newLine();
            }
        }
        
        // Create metadata file
        try (BufferedWriter writer = Files.newBufferedWriter(inbox.resolve("orders.json.metadata"))) {
            writer.write("region:orders\n");
            writer.write("ttl:86400\n");
            writer.write("key_field:order_id\n");
        }
        
        System.out.println("Files created. Kuber will process them on next scan cycle.");
    }
}

Replication Support

Automatic Replication: Data loaded via autoload is automatically replicated to secondary nodes when replication is enabled. The autoload service uses the standard CacheService.jsonSet() method which triggers replication.

Best Practices

  • Always test with small files first - Verify your metadata format before processing large files
  • Use meaningful key fields - Choose a field with unique values (IDs, SKUs, etc.)
  • Set appropriate TTLs - Use -1 for permanent data, positive values for temporary data
  • Monitor the outbox - Check for ERROR files and investigate failures
  • Use CSV for simple data - CSV is faster to parse than JSON
  • Use JSONL for complex structures - When you have nested objects or arrays
  • Consider file sizes - Large files may take time; use max-records-per-file to limit

Kuber supports multiple persistence backends, allowing you to choose the storage solution that best fits your deployment needs. The persistence layer is pluggable and can be changed via configuration without code modifications.

Note: The default persistence backend is MongoDB. To switch backends, set the kuber.persistence.type property in your application.properties file and restart Kuber.

Replication

Kuber supports Primary/Secondary replication using ZooKeeper for coordination.

Configuration

# Enable replication
kuber.replication.enabled=true
kuber.replication.zookeeper.connect-string=localhost:2181
kuber.replication.zookeeper.base-path=/kuber
kuber.replication.zookeeper.session-timeout-ms=30000

Behavior

  • PRIMARY - Accepts reads and writes, replicates to secondaries
  • SECONDARY - Read-only, receives replicated data
  • Automatic Failover - Secondary becomes primary if primary fails
